# 하비루프 React 잔여 마이그레이션 계획

## 1. 문서 목적

이 문서는 2026-07-19 기준 React로 전환된 화면을 점검하고, 기능 동등성과 운영 전환을 완료하기 위해 남은 작업을 정의한다.

기준 문서의 우선순위는 다음과 같다.

1. [`API_SPEC.md`](./API_SPEC.md)의 확정된 API 계약
2. 이 문서의 잔여 작업 계획
3. [`REACT_MIGRATION_PLAN.md`](./REACT_MIGRATION_PLAN.md)의 전체 계획
4. 현재 레거시 HTML/JavaScript 동작

기존 마이그레이션 규칙을 그대로 적용한다.

- JavaScript와 JSX만 사용한다.
- API 경로와 응답 필드는 추측하지 않는다.
- 서버 데이터는 `fetch` 기반 service와 `useState`, `useEffect`로 관리한다.
- 한 번에 한 기능을 구현하고 lint, test, build, 사용자 시나리오를 검증한다.
- 레거시 파일 삭제는 기능 동등성과 rollback 가능성을 확인한 뒤 별도 단계에서 수행한다.

## 3. 선행 API 계약 확정

이 단계는 임의 필드 구현을 방지하기 위한 P0 작업이다. 확인한 계약은 먼저 `API_SPEC.md`에 반영한다.

### 3.1 좋아요 초기 상태

확인할 내용:

- `GET /articles/:articleId` 응답에 현재 사용자의 좋아요 여부가 포함.
- 필드명이 `likedByMe`이다.
- 좋아요 POST/DELETE 성공 응답에 변경된 count 또는 상태가 포함되지 않는다.


### 3.2 소유권 식별자

현재 React 구현은 게시글과 댓글의 작성자 닉네임을 로그인 사용자 닉네임과 비교한다. 닉네임은 권한 식별자로 안전하지 않으므로 다음 중 하나를 확정한다.

- 응답에 작성자의 `userId` 제공

대상 응답:

- `GET /users/me`
- `GET /articles/:articleId`
- `GET /articles/:articleId/comments`
- 댓글 생성 응답

### 3.3 프로필 이미지 업로드 — 계약 반영 완료

확정 및 반영한 내용:

- 회원가입 전에는 `/users/me/profile-image`를 access token 없이 호출한다.
- 로그인 후 프로필 변경에서는 인증 요청을 사용한다.
- 가입용과 수정용 endpoint는 분리하지 않는다.
- 파일당 10MB 제한을 선택 단계에서 검증한다.
- 업로드 성공 후 회원가입이나 프로필 저장이 실패하면 같은 URL을 재사용한다.
- preview object URL은 파일 변경과 unmount 시 해제한다.

서버가 허용하는 MIME과 해상도는 문서에 확정되지 않았으므로 브라우저 파일 선택은 `image/*`로 제한하되 별도의 MIME 목록을 추측해 검증하지 않는다.

### 3.4 사용자 오류와 세션 정책

확인할 내용:

- 회원가입 이메일 중복 오류 `409`
- 회원가입 및 프로필 수정 닉네임 중복 오류 `409`
- 오류 응답의 field 정보 구조
- 비밀번호 변경 성공 후 재로그인 정책은 반영 완료

### 3.5 P0 완료 기준

- 위 계약이 `API_SPEC.md`에 확정 상태로 기록되어 있다.
- 좋아요, 소유권, 프로필 이미지 구현에 추측 필드가 필요하지 않다.
- 정상/오류 응답 fixture 또는 OpenAPI 예시를 확보한다.

## 4. 단계별 구현 계획

### 1단계. 게시글 좋아요 — 완료

작업:

1. 상세 mapper에 확정된 좋아요 상태를 추가한다.
2. `articleService.js`에 좋아요 추가/취소 함수를 구현한다.
3. `useArticleDetail`이 좋아요 상태와 count를 변경할 수 있도록 action을 제공한다.
4. `PostStats`에 접근 가능한 좋아요 버튼과 요청 상태를 연결한다.
5. 요청 중 중복 클릭을 막고 성공 후에만 상태와 count를 갱신한다.
6. 실패하면 기존 상태를 유지하고 재시도 가능한 오류를 표시한다.
7. 409 처리 방식은 확정된 서버 계약을 따른다.

필수 테스트:

- 좋아요하지 않은 상태에서 POST 성공
- 좋아요한 상태에서 DELETE 성공
- network/4xx/5xx 실패 시 기존 상태 유지
- 요청 중 중복 클릭 방지
- count가 0 미만으로 내려가지 않음
- 상세 mapper가 서버의 초기 좋아요 상태를 보존함

완료 기준:

- 새로고침 직후 서버 상태와 버튼 상태가 일치한다.
- 성공/실패 후 좋아요 상태와 count가 서로 어긋나지 않는다.

### 2단계. 게시글·댓글 소유권 판별 — 완료

작업:

1. 사용자, 게시글, 댓글 mapper에 확정된 소유권 필드를 추가한다.
2. 닉네임 비교 로직을 제거한다.
3. 작성자에게만 게시글 수정/삭제와 댓글 수정/삭제 action을 표시한다.
4. UI가 action을 숨겨도 서버의 403 오류 처리를 유지한다.
5. 프로필 닉네임 변경 후에도 기존 작성물의 소유권 표시가 유지되는지 확인한다.

필수 테스트:

- 본인 게시글과 댓글에 action 표시
- 다른 사용자의 게시글과 댓글에 action 미표시
- 같은 닉네임을 가진 다른 사용자에게 action 미표시
- 직접 요청한 수정/삭제가 403이면 기존 화면 유지

완료 기준:

- 모든 소유권 판단이 불변 사용자 식별자 또는 서버의 `isOwner`를 사용한다.
- 닉네임 변경과 중복 닉네임이 권한 UI에 영향을 주지 않는다.

### 3단계. 프로필 이미지 업로드 — 완료

작업:

1. 확정된 endpoint와 인증 정책으로 image service를 구현한다.
2. 공통 프로필 이미지 picker를 만든다.
3. 파일 형식과 크기를 선택 시점에 검증한다.
4. object URL preview를 제공하고 파일 변경/unmount 시 URL을 해제한다.
5. 회원가입에서는 업로드 URL을 `profileImageUrl`로 전송한다.
6. 프로필 수정에서는 기존 이미지 유지, 교체, 필요하면 삭제 상태를 구분한다.
7. 업로드 성공 후 저장 실패 시 URL을 재사용하여 자동 재업로드를 막는다.
8. 프로필 수정 성공 후 AuthContext와 헤더 이미지를 즉시 갱신한다.

필수 테스트:

- 이미지 없이 회원가입
- 유효한 이미지 preview와 업로드
- 허용되지 않은 형식/크기 거부
- 업로드 실패 시 폼 입력 유지
- 저장 실패 후 같은 업로드 URL 재사용
- 프로필 이미지 변경 후 헤더 즉시 반영
- object URL cleanup

완료 기준:

- 회원가입과 프로필 수정 양쪽에서 확정된 인증 정책으로 업로드가 동작한다.
- 실패가 가입/수정 성공으로 잘못 처리되지 않는다.

### 4단계. 오류 메시지와 인증 후속 정책

작업:

1. 이메일·닉네임 중복 오류를 확정된 field에 연결한다.
2. 일반 오류와 field 오류를 구분하되 입력값을 유지한다.
3. 비밀번호 변경 후 세션 정책을 적용한다.
4. 세션 만료 정책이면 명시적으로 인증 정보를 제거하고 로그인 화면으로 이동한다.
5. 세션 유지 정책이면 성공 toast와 현재 보호 화면을 유지한다.

필수 테스트:

- 이메일 중복 오류가 이메일 field에 표시됨
- 닉네임 중복 오류가 닉네임 field에 표시됨
- 알 수 없는 오류는 form 공통 오류로 표시됨
- 비밀번호 변경 후 확정된 인증 상태 전이

완료 기준:

- 서버 오류 code가 사용자에게 잘못된 field 메시지로 표시되지 않는다.
- 비밀번호 변경 후 화면과 인증 상태가 서버 정책과 일치한다.

## 5. E2E 및 전환 검증 계획

### 5.1 Playwright 기반 구성

Playwright와 `test:e2e` script를 추가했으며 route mocking으로 다음 핵심 흐름을 자동화했다.

- 미인증 상세 deep link → 로그인 → 원래 URL 복귀
- 좋아요 추가와 동일 닉네임·다른 `userId` 소유권 검증
- 비밀번호 변경 후 인증 해제와 로그인 이동

후속 E2E에서는 다음 구성을 확장한다.

- Playwright 설정과 브라우저별 기본 프로젝트
- 테스트용 API base URL 및 사용자 계정 주입 방식
- 독립적으로 생성하고 정리할 수 있는 테스트 데이터 규칙
- 실패 시 screenshot, trace, video 보존 정책
- CI에서 실행할 `test:e2e` script

### 5.2 필수 E2E 흐름

1. 미인증 상세 deep link → 로그인 → 원래 URL 복귀
2. 로그인 및 로그아웃
3. 회원가입 성공/중복/프로필 이미지 선택 흐름
4. 게시글 목록 첫 로딩, 추가 로딩, 오류 재시도
5. 게시글 작성 → 상세 → 수정 → 삭제
6. 좋아요 추가/취소와 새로고침 후 상태 유지
7. 댓글 작성/수정/삭제
8. 대댓글 작성/취소와 부모 아래 표시
9. 프로필 수정 후 헤더 반영
10. 비밀번호 변경과 확정된 세션 처리
11. 회원 탈퇴 후 보호 route 접근 차단
12. 존재하지 않는 route와 게시글 404 구분

### 5.3 접근성·시각 검증

- 데스크톱과 모바일 viewport에서 주요 route screenshot 비교
- 키보드만으로 메뉴, gallery, dialog, 댓글 편집/답글 사용
- dialog 열기/닫기 후 focus 복귀
- form error의 `aria-invalid`, `aria-describedby`, live region 확인
- 이미지 fallback과 긴 제목/내용/닉네임 레이아웃 확인

### 5.4 staging 검증

- `VITE_API_BASE_URL` 주입 확인
- Vite/배포 origin과 API 사이의 CORS 확인
- refresh cookie의 SameSite, Secure, Domain 확인
- 모든 SPA route 직접 접근 및 새로고침 확인
- API 401 → refresh → 원 요청 재시도 확인
- production build 정적 asset 경로 확인

## 6. 레거시 제거와 전환

React에서 사용하지 않는 HTML/JavaScript는 `legacy/`로 격리했고, 미사용 `RoutePlaceholderPage`는 제거했다. `css/`와 `assets/`는 React가 계속 사용하므로 유지한다.

### 제거 후보

- `pages/auth/*.html`
- `pages/posts/*.html`
- `pages/user/*.html`
- `js/pages/**`
- React에서 사용하지 않는 `js/common/**`, `js/constants/**`
- 미사용 `src/pages/RoutePlaceholderPage.jsx`

`css/`와 `assets/`는 React가 계속 사용하므로 사용 여부를 파일별로 확인하기 전에는 제거하지 않는다.

### 제거 절차

1. `rg`로 React 코드의 레거시 import/경로 참조가 없는지 확인한다.
2. 모든 E2E 시나리오와 staging 검증을 통과한다.
3. 레거시 제거 파일 목록을 리뷰한다.
4. rollback용 태그 또는 배포 artifact를 보존한다.
5. 승인된 파일만 제거한다.
6. lint, unit/component test, E2E, build를 다시 실행한다.
7. 배포 후 인증 실패율, API 오류율, 주요 사용자 흐름을 관찰한다.

완료 기준:

- React 진입점만으로 모든 사용자 흐름이 동작한다.
- 배포 서버의 deep-link rewrite가 동작한다.
- 레거시 제거 후에도 CSS와 asset 누락이 없다.
- 문제가 생겼을 때 이전 artifact로 되돌릴 수 있다.

| 6 | Playwright 기반과 계약 독립 E2E 추가 | 기반 및 핵심 3개 완료, 시나리오 확장 필요 |
| 7 | 계약 의존 E2E와 staging 검증 | 백엔드/환경 필요 |
| 8 | 레거시 제거 후보 검토 및 승인 | 전체 회귀 통과 후 가능 |
| 9 | 레거시 제거와 운영 전환 | 승인 후 가능 |

계약 확인을 기다리는 동안에는 Playwright 기반 구성, 현재 확정된 인증/목록/게시글/댓글 흐름의 E2E, 미사용 파일 분석을 병행할 수 있다.

## 8. 최종 Definition of Done

- 계획된 모든 route와 기능이 React에서 동작한다.
- 좋아요 상태가 서버와 일치한다.
- 소유권 UI가 닉네임이 아닌 확정된 식별자로 판별된다.
- 회원가입과 프로필 수정의 이미지 업로드가 동작한다.
- 서버 field 오류와 비밀번호 변경 세션 정책이 반영된다.
- lint, unit/component test, Playwright E2E, production build가 통과한다.
- 모바일/데스크톱 및 키보드 접근성 검증을 통과한다.
- staging에서 CORS, cookie, 환경 변수, SPA rewrite가 검증된다.
- 승인된 레거시 파일이 제거되고 rollback 절차가 준비되어 있다.
- `API_SPEC.md`에 구현을 막는 `확정 필요` 항목이 남아 있지 않다.
